# MOC

A Map of Content is a navigation page that lists and links the pages in a scope — the established PKM term for what this plugin implements as folder notes and the vault index.

## Definition

MOC (Map of Content) is an established personal knowledge management term for a page whose primary purpose is navigation: it lists, links, and briefly summarises the pages within its scope rather than presenting novel content of its own. The concept comes from the Zettelkasten and Obsidian PKM community; the plugin adopts the term and its standard conventions without inventing a new one.

The plugin implements two scopes of MOC:

- **Per-folder MOC** — the folder note (`wiki/<topic>/<topic>.md`, `type: index`), which lists all pages in its topic branch under grouped section headings. Every topic folder contains exactly one folder note with this role.
- **Vault MOC** — `wiki/index.md`, which lists all top-level topic folder notes and serves as the graph's ROOT hub. It is the single entry-point node from which every topic island is reachable.

Both scopes use the same `type: index` frontmatter and the same piped-basename wikilink convention for their `children:` and `child_indexes:` entries.

## Key Principles

**Navigation, not content.** A MOC page contains headings and `- [[page-basename|Page Title]] — one-line summary` entries, not prose claims. It is regenerated by the polish agent after ingest; hand-editing is merge-safe but the automation should be treated as authoritative.

**`parent:` spine integration.** Every non-root page carries a `parent:` wikilink pointing to its containing folder note (its MOC). This makes the MOC hierarchy traversable both through the filesystem and through the wikilink graph. The analyst's MOC descent (C1) follows `parent:` links downward from the vault MOC to collect the working set for a query.

**`aliases` for discovery.** Folder notes carry `aliases:` with slug, title-case, and abbreviation variants of the topic name so search and autocomplete can find them by any form. Aliases do not affect wikilink resolution; they are purely a discovery aid.

**Regenerable cache.** The vault MOC and folder notes are regenerated deterministically by the polish agent from the filesystem and frontmatter. If an index drifts from reality (a page added but not listed), `lint` reports it and the curator corrects it. Because they are regenerable, a lost or mangled MOC is not data loss.

**ROOT hub role.** `wiki/index.md` carries the alias `ROOT` and is treated as the graph's single entry-point node by Obsidian's graph color configuration. Every topic island is reachable from ROOT through the vault MOC's `child_indexes:` entries, so the graph never splits into floating islands with no anchor.

## Examples

The vault MOC at `wiki/index.md` contains a `## Topics` section with entries like `- [[agents|Agents]] — pages covering the eight Layer 3 executor agents`. Clicking that link navigates to the agents folder note, which in turn lists every page in `wiki/agents/`.

An analyst querying "what pages exist about the ingest pipeline?" performs MOC descent: reads `wiki/index.md`, finds the folder note for the relevant topic, reads the folder note's `children:`, and loads those pages as the working set.

## Related Concepts

MOC relates to the folder note (the per-folder MOC implementation), the ROOT hub (the vault MOC's role in the graph), MOC descent (the C1 retrieval path that traverses MOC links), the polish agent (which regenerates MOCs after ingest), and the `index` page type (the frontmatter type used by all MOC pages).
